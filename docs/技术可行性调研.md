# 技术可行性调研（Technical Feasibility Research）

## 1. 静态分析与语法解析：工具选型对比

为了实现 1 周内研发出可运行的 MVP 工程目标，RepoMind 必须在底层工具链和算法路径上进行高可行性的技术抉择。

### 1.1 Python 静态分析工具对比

#### 1.1.1 py-tree-sitter

**特点**：
- 支持增量解析，采用 C 语言编写
- 具备极强的语法容错（Error-tolerant）能力
- 即便代码残缺、未配置完整依赖、甚至存在语法错误，依然能够提取出高价值的 AST 树
- 纳秒级极速解析，拥有绝佳的语法容错能力

**适用场景**：
- 构建轻量级本地 CLI 工具的首选
- 能够在秒级内解析大型 Python 源文件

**可行性评估**：⭐⭐⭐⭐⭐（高度可行）

#### 1.1.2 Jedi

**特点**：
- Python 静态分析库，专为代码补全和类型推断设计
- 支持跳转定义、类型推断、自动补全
- 轻量级，无需完整编译
- 广泛用于 VS Code、Vim 等编辑器插件

**适用场景**：
- Python 项目的类型推断和符号解析

**可行性评估**：⭐⭐⭐⭐⭐（高度可行）

#### 1.1.3 Pyright

**特点**：
- 微软开发的 Python 类型检查器
- 实现了完整的类型推断算法
- 支持 Type Hints、泛型、协议等高级特性
- 性能优异，比 mypy 快 10-50 倍

**适用场景**：
- 需要深度类型推断的项目

**可行性评估**：⭐⭐⭐⭐（可行，但 MVP 阶段可简化）

#### 1.1.4 ast (Python 内置)

**特点**：
- Python 标准库内置的 AST 解析模块
- 零额外依赖
- 功能有限，不支持增量解析
- 性能较 Tree-sitter 慢

**适用场景**：
- 简单的语法分析任务

**可行性评估**：⭐⭐⭐（可用，但不推荐作为主方案）

### 1.2 技术选型结论

**首期采用 py-tree-sitter + Jedi 双引擎架构**

- **py-tree-sitter**：作为通用 AST 符号提取底座，负责快速解析和语法容错
- **Jedi**：作为类型推断引擎，处理 Python 动态类型的静态分析

**针对 Python 动态类型的挑战，采用渐进式类型推断策略**：

1. **显式类型提示（置信度 0.95）**：直接提取 `def func(x: int) -> str` 中的类型信息
2. **赋值推断（置信度 0.70）**：通过分析 `x = get_user()` 推断变量类型
3. **函数返回值推断（置信度 0.60）**：分析函数体推断返回类型
4. **鸭子类型匹配（置信度 0.40）**：通过方法调用反推对象类型

**目标**：在无需类型提示的情况下，通过多策略级联实现 70% 以上的类型推断准确率。

## 2. 关系图存储：方案对比

### 2.1 Neo4j

**特点**：
- 企业级分布式图数据库
- 支持强大的 Cypher 复杂拓扑查询和图算法库（GDS）
- 部署过于繁重，需后台常驻 JVM 服务并耗费数 GB 的常驻运行内存
- 将百万行 Linux 内核全量入库需要 4 个小时以上
- 难以打包装箱进本地 CLI 中运行

**适用场景**：
- 大规模生产环境的图数据存储

**可行性评估**：⭐⭐（不适用）
- 部署复杂，资源消耗大

### 2.2 SQLite + NetworkX

**特点**：
- SQLite 充当磁盘持久化层
- NetworkX 充当内存高速图计算层
- NetworkX 原生支持 Louvain 社群聚类、PageRank 中心度评估、BFS 拓扑扩展等算法
- 通过 Aho-Corasick 高效的模式匹配，在 3 分钟内即可完成 2800 万行 Linux 内核级图谱构建
- 查询延迟处于微秒级

**适用场景**：
- 本地优先（Local-first）和零配置（Zero-configuration）的产品哲学

**可行性评估**：⭐⭐⭐⭐⭐（高度可行）

### 2.3 技术选型结论

**选择 SQLite + NetworkX 组合**

满足本地优先（Local-first）和零配置（Zero-configuration）的产品哲学。在未来扩展中，可通过轻量级扩展 GraphQLite 引入 Cypher 查询支持，实现极速单文件 GraphRAG 闭环。

## 3. Agent 协作范式：架构对比

### 3.1 Explorer-Solver 双 Agent

**特点**：
- Explorer 专门负责在图谱中跳转、收集证据、剪枝垃圾文件
- Solver 在一个绝对干净、零检索干扰的上下文（Evidence Bundle）中进行高难度的逻辑编写与 Root Cause 分析
- 能有效规避多轮对话下的"自省混乱"与"瀑布式降级"（Waterfall Degradation）

**优势**：
- 关注点分离，提高推理质量
- 减少上下文污染
- 支持复杂的多步骤任务

### 3.2 Agentless 极简硬编码管道

**特点**：
- 伊利诺伊大学提出的 Agentless 方案
- 摒弃了让 LLM 自行决定"下一步行动"的自主规划（Autonomous Planning）权力
- 采用"定位 → 修复 → 校验"硬编码确定性工作流
- 极大地降低了 Token 消耗和自省死锁
- 在 SWE-bench Verified 上取得了 50% 以上的优秀解决率

**优势**：
- 流程确定性强，可预测
- Token 消耗低
- 避免无限循环和死锁

### 3.3 技术选型结论

**采取"折中"的最优解**：

- **底层**：采用 Agentless 式的强约束控制流管道（Linear State Machine）作为流程骨架，杜绝多 Agent 之间无休止的自然语言讨论死锁
- **核心节点**：在"定位"与"根因深度推理"两个核心状态节点内部，拉起专属的 Explorer（提取代码）与 Solver（生成根因说明与修复补丁）工具
- **设计原则**：将高内聚任务模块化，确保一星期内代码编写可控

## 4. 关键技术算法

### 4.1 Python 动态类型推断算法（Progressive Type Inference）

针对 Python 动态类型的挑战，系统采用渐进式多策略级联推断算法：

#### 4.1.1 显式类型提示提取（置信度 C₁ = 0.95）

直接从 AST 中提取 Python Type Hints：

```python
# 函数签名类型提示
def process(user: User) -> Order: ...  # 提取: User -> Order

# 变量类型提示
name: str = "hello"  # 提取: name: str

# 类属性类型提示
class Config:
    timeout: int = 30
    debug: bool = False
```

**置信度分析**：Type Hints 是开发者显式声明的类型信息，准确率最高。

#### 4.1.2 Import 映射绑定（置信度 C₂ = 0.90）

解析 Python Import 语句，建立符号到模块的映射：

```python
# 绝对导入
from datetime import datetime  # datetime.datetime
import os.path  # os.path

# 相对导入
from .models import User  # 当前包.models.User
from ..utils import helper  # 父包.utils.helper
```

**Python 特有挑战**：
- 相对导入需要计算包层级
- `__init__.py` 隐式导出
- 通配符导入（`from module import *`）

#### 4.1.3 赋值语句类型推断（置信度 C₃ = 0.70）

通过分析赋值语句右侧表达式推断变量类型：

```python
# 构造函数调用
user = User()  # 推断: user: User

# 函数返回值
order = create_order(user)  # 推断: order: Order (需分析 create_order 返回类型)

# 链式调用
result = user.get_orders().filter(active=True)  # 推断: result: QuerySet[Order]
```

**推断策略**：
1. 分析赋值右侧的表达式类型
2. 追踪函数返回值类型
3. 处理链式调用的类型传播

#### 4.1.4 self 参数类型推断（置信度 C₄ = 0.85）

Python 类方法中的 `self` 参数隐含了类型信息：

```python
class UserService:
    def get_user(self, user_id: int) -> User:
        # self 的类型是 UserService
        # 可以推断 self.get_user() 返回 User
        ...
```

**应用场景**：
- 类内方法调用的类型绑定
- 实例属性访问的类型推断

#### 4.1.5 鸭子类型匹配（置信度 C₅ = 0.40）

当无法通过上述策略推断时，基于方法调用反推对象类型：

```python
def process(obj):
    obj.do_something()  # 鸭子类型推断
    # 全局搜索所有有 do_something 方法的类
    # 返回候选列表，按使用频率排序
```

**匹配策略**：
1. 全局搜索有相同方法名的类
2. 分析方法签名兼容性
3. 结合上下文使用模式
4. 按使用频率和模块相关性排序

#### 4.1.6 Jedi 类型推断兜底（置信度 C₆ = 0.60）

调用 Jedi 静态分析引擎进行深度类型推断：

```python
import jedi

script = jedi.Script(source_code)
# 获取变量类型
definitions = script.infer(line, column)
# 获取函数签名
signatures = script.get_signatures(line, column)
```

**优势**：
- 支持复杂的类型推断场景
- 处理泛型、协议等高级特性
- 社区维护，持续更新

### 4.2 Python 调用图构建策略

针对 Python 动态特性，采用多策略调用图构建：

#### 4.2.1 显式类型调用（高置信度）

```python
user = get_user()
user.get_name()  # 如果知道 user: User，则绑定到 User.get_name()
```

#### 4.2.2 self 调用（高置信度）

```python
class Service:
    def process(self):
        self.helper()  # 绑定到 Service.helper()
```

#### 4.2.3 全局函数调用（中置信度）

```python
from utils import helper
helper()  # 绑定到 utils.helper
```

#### 4.2.4 鸭子类型推断（低置信度）

```python
def process(obj):
    obj.do_something()  # 全局搜索有 do_something 方法的类
```

### 4.3 混合检索与上下文工程

**混合检索流动逻辑**：

1. BM25 关键字检索与基于代码专用 Embedding 模型的语义检索并行运行
2. 在初次匹配获得种子节点（Seed Nodes）后
3. 通过静态代码图谱（SQLite+NetworkX）触发深度优先（DFS）或广度优先（BFS）的"图谱向外拓扑扩展"（Graph Expansion）
4. 扩展算法沿 CALLS、INHERITS 和 IMPORTS 边向外延伸两跳（2-hop neighbors）
5. 自动抓取种子节点所依赖的方法声明、接口规范以及上游调用链
6. 将局部碎片代码拼装回完整的"物理代码上下文树"

**上下文压缩工程**：

- **符号增量账本（Symbol Delta Ledger, SDL）**：仅将代码变动部分（Delta）以及变动行关联的最窄调用链范围记录为紧凑的账本格式
- **代码骨架化剪枝（AST-based Skeleton Pruning）**：通过 AST 将外部关联文件的具体函数体内容全部抹除，仅保留类签名、字段声明、方法签名

## 5. 可行性总结

| 技术维度 | 推荐方案 | 可行性评估 | 关键优势 |
|---------|---------|-----------|---------|
| 语法解析 | py-tree-sitter (Python) | ⭐⭐⭐⭐⭐ | 纳秒级解析，语法容错强 |
| 类型推断 | Jedi + 渐进式推断算法 | ⭐⭐⭐⭐ | 支持 Python 动态类型，70%+ 准确率 |
| 图存储 | SQLite + NetworkX | ⭐⭐⭐⭐⭐ | 零配置，本地优先，微秒级查询 |
| Agent 架构 | Explorer-Solver + 线性管道 | ⭐⭐⭐⭐⭐ | 确定性强，避免死锁 |
| 检索策略 | 混合检索 + 图拓扑扩展 | ⭐⭐⭐⭐⭐ | 精准召回，节省 90%+ Token |

**结论**：所有关键技术路径均具备高度可行性，能够在 1 周内完成 MVP 开发。Python 动态类型带来的挑战通过渐进式类型推断算法（Type Hints + 赋值推断 + 鸭子类型匹配）得到有效解决。
