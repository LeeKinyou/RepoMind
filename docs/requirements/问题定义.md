# 问题定义（Problem Definition）

## 1. 核心技术冲突

在大规模软件工程开发中，代码库规模的指数级增长与模型上下文窗口限制、推理噪声之间的矛盾，构成了当前 AI 辅助软件工程（AIDE）的核心技术冲突。

### 1.1 上下文窗口有限 vs 几十万行代码的物理规模

尽管 2026 年的主流前沿大模型（如 Claude Opus 4.8、Gemini 3.1 Pro 等）宣称其拥有百万乃至千万级别的上下文窗口，但在实际工程中，将整个几十万行甚至数百万行的代码仓库一次性塞入 Context 存在严重局限性：

- **高昂的 Token 成本**：处理大型代码库需要消耗大量 Token，导致 API 调用成本激增
- **巨大的网络延迟**：传输海量代码数据造成显著的响应延迟
- **"中间信息丢失"（Lost in the Middle）现象**：模型在海量噪音信息中丢失对关键代码逻辑的注意力

### 1.2 纯向量检索（Vector-only RAG）的"语义迟钝"

传统的向量相似度匹配算法无法精准感知代码控制流和数据流的边界。代码中极小的语法修改（例如添加一个非操作符 `!`、执行类型强转、或修改装饰器配置）会彻底改变程序执行结果，但在向量空间中其余弦相似度的偏移极其微弱：

$$Cosine(u, v) = \frac{u \cdot v}{|u| |v|} \approx 0.99$$

**Python 动态类型特性加剧了这一问题**：

```python
# 看起来相似，但语义完全不同
def process(user: User) -> Order: ...  # 类型明确
def process(data) -> dict: ...          # 类型模糊，可能是任何东西

# 鸭子类型导致同名方法语义不同
class UserService:
    def validate_token(self, token): ...  # JWT 验证
class PaymentService:
    def validate_token(self, token): ...  # 支付令牌验证
```

这会导致 RAG 系统频繁召回具有相似自然语言注释但完全无关的"同名不同义"方法（如多服务下同名的 `validate_token()`），产生严重的"幻觉"和错误的修复补丁。

### 1.3 仓库级结构理解（Repository Structure Understanding）的缺失

### 1.3 仓库级结构理解（Repository Structure Understanding）的缺失

传统的检索手段割裂了代码文件的物理依赖。代码仓库的本质是一张由继承（INHERITS）、实现（IMPLEMENTS）、调用（CALLS）和导入（IMPORTS）等强符号关系交织而成的有向图。

**Python 的继承和导入机制更加复杂**：

```python
# Python 多继承和 MRO（方法解析顺序）
class Service(BaseService, MixinA, MixinB):
    def process(self): ...

# 相对导入增加了依赖追踪难度
from .models import User
from ..utils import helper
from ...core.exceptions import ValidationError

# 鸭子类型使得接口关系隐式化
class PaymentProcessor:  # 没有显式实现接口
    def process_payment(self, amount): ...  # 但实现了 PaymentInterface 的方法
```

缺乏图谱拓扑关系的智能体只能在盲目的目录遍历（File Crawling）和全局文本搜索（Grep）中循环挣扎，无法完成多跳跨文件依赖推理。

### 1.4 长期仓库记忆（Repository Memory）的缺失

目前的 SE Agent 绝大多数运行在"单次会话"模式下。每次运行都必须从头扫描或临时检索，无法将历史架构设计决策（ADR）、项目特定命名规范及特定规避 Gotchas 沉淀为持久的知识资产（如 Claude Code 使用的 memory.md 规范）。

## 2. 目标用户与痛点场景（User & Scenarios）

本平台主要服务于以下三类核心研发人员：

### 2.1 新人接手项目（Onboarding）

新开发者面对百万行历史遗留系统（Legacy Codebase）时，难以建立全局架构心智模型，手动追踪调用链路时间成本极高。

**痛点**：

- 缺乏对系统整体架构的快速认知
- 难以理解历史代码的设计意图和业务逻辑
- 跨文件依赖关系复杂，手动追踪效率低下

### 2.2 开源仓库阅读与架构分析

研究人员或系统架构师需要快速理清大型开源项目的核心生命周期、Hub 节点和子社区分布，而无需本地搭建繁琐的编译运行环境。

**痛点**：

- 需要快速理解复杂系统的架构设计
- 避免繁琐的环境配置和依赖安装
- 需要可视化工具辅助分析

### 2.3 复杂 Bug 精准定位与根因调查（RCA）

SRE 或 SDE 在生产环境发生 crash 并拿到 Stack Trace 时，由于分布式链路跳转和多文件依赖，排查 proximate（直接原因）与 systemic（系统原因）的过程耗时耗力。

**痛点**：

- 分布式系统故障定位困难
- 多文件依赖导致根因分析复杂
- 手动排查耗时，影响系统恢复时间

***

## 3. 问题总结

RepoMind 项目旨在解决以下核心问题：

1. **上下文爆炸问题**：如何在有限的 LLM 上下文窗口中有效处理大型代码库
2. **语义理解不足**：如何超越简单的向量相似度，实现精准的代码语义理解
3. **结构感知缺失**：如何构建代码间的拓扑关系图，支持多跳依赖推理
4. **记忆持久化**：如何将项目知识沉淀为可复用的持久化资产
5. **效率提升**：如何大幅降低开发者在代码理解和故障排查上的时间成本

**Python 动态类型带来的额外挑战**：

6. **类型推断难题**：如何在无显式类型声明的情况下，准确推断变量和函数返回值类型
7. **鸭子类型处理**：如何识别隐式接口实现，构建准确的调用图
8. **多继承复杂性**：如何处理 Python 的多继承和 MRO（方法解析顺序）
9. **相对导入解析**：如何正确处理 Python 的相对导入机制

通过构建面向大型代码仓库的 Repository Intelligence Platform，RepoMind 致力于为开发者提供一个高效、精准、智能的代码理解和根因分析工具。首期聚焦 Python 语言，通过渐进式类型推断算法解决动态语言的静态分析难题。
