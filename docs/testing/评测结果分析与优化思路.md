# RepoMind 当前问题总结：Symbol-only 异常与 Docker Sandbox Fallback

## 1. 当前背景

RepoMind 在完成一轮检索与评测优化后，`full` 模式的指标已经有明显提升：

```json
{
  "keyword_only": {
    "top1_rate": 0.54,
    "top3_rate": 0.78,
    "func_rate": 0.505,
    "latency": 0.0057
  },
  "symbol_only": {
    "top1_rate": 0.0,
    "top3_rate": 0.0,
    "func_rate": 0.0,
    "latency": 0.0031
  },
  "hybrid": {
    "top1_rate": 0.42,
    "top3_rate": 0.68,
    "func_rate": 0.46,
    "latency": 0.1429
  },
  "full": {
    "top1_rate": 0.76,
    "top3_rate": 0.9,
    "func_rate": 0.6017,
    "latency": 0.0219
  }
}
```

从整体看，`full` 模式已经达到：

| Mode | Top-1 File Hit | Top-3 File Hit | Function Hit | Latency |
| ---- | -------------: | -------------: | -----------: | ------: |
| full |          76.0% |          90.0% |        60.2% |  0.022s |

这说明 RepoMind 的完整检索链路已经具备较强的文件级定位能力。

但是当前仍然有两个明显问题需要修复：

1. `symbol_only` 指标全为 0，明显异常。
2. 运行过程中出现 `Docker daemon is not available. Falling back to subprocess sandbox.`，说明 Docker 沙箱不可用时的 fallback 逻辑还不够清晰。

---

## 2. 问题一：symbol_only 得分异常为 0

当前结果：

```json
"symbol_only": {
  "top1_rate": 0.0,
  "top3_rate": 0.0,
  "func_rate": 0.0,
  "latency": 0.0030546855926513673
}
```

这个结果大概率不是“symbol 检索能力差”，而是 symbol-only 检索链路存在工程问题。

正常情况下，即使 benchmark 中很多 query 是自然语言描述，不直接包含函数名，`symbol_only` 也不应该完全为 0。比较合理的情况应该是：

| Metric         | Expected Range |
| -------------- | -------------: |
| Top-1 File Hit |      20% - 45% |
| Top-3 File Hit |      40% - 65% |
| Function Hit   |      30% - 60% |

当然，这只是预期范围，不是硬性目标。真正重点是：`symbol_only` 应该能证明 AST symbol index 确实被构建、检索、返回并参与评测。

---

## 3. symbol_only 可能异常的原因

### 3.1 Symbol Index 没有构建成功

可能原因：

1. 没有扫描到目标仓库目录。
2. 文件过滤规则排除了 `.py` 文件。
3. Windows 路径处理异常。
4. AST parse 报错后被静默忽略。
5. benchmark 使用的是 host repo，但 symbol index 构建的是当前 RepoMind 自身。
6. 构建成功但没有保存到 retriever 使用的数据结构中。

需要在 benchmark 或 index 构建阶段输出诊断信息：

```text
[SymbolIndex] files_scanned=?
[SymbolIndex] python_files=?
[SymbolIndex] symbols_extracted=?
[SymbolIndex] functions=?
[SymbolIndex] classes=?
[SymbolIndex] methods=?
```

如果 `symbols_extracted=0`，说明问题在 index 构建阶段。

---

### 3.2 Symbol result 没有映射回 file_path

benchmark 的核心指标是：

```text
Top-1 File Hit
Top-3 File Hit
Function Hit
```

因此即使是 `symbol_only`，返回结果也必须包含文件路径。

如果当前 symbol-only 只返回：

```text
qualified_name = "RepoIndexer.build_index"
```

但没有返回：

```text
file_path = "src/retrieval/indexer.py"
```

那么 evaluator 就无法计算文件级命中率，最终可能全是 0。

正确的 symbol result 至少应该包含：

```python
file_path: str
symbol_name: str
qualified_name: str
start_line: int
end_line: int
score: float
source: str = "symbol_only"
```

---

### 3.3 Evaluator 不识别 symbol result

即使 symbol-only 返回了结果，如果 result schema 和 evaluator 预期不一致，也会导致全部评测失败。

例如 evaluator 只读取：

```python
result["path"]
```

但 symbol result 里叫：

```python
result["file_path"]
```

就会导致命中率全为 0。

需要统一 RetrievalResult schema，或者在 evaluator 中兼容不同字段：

```python
file_path = result.file_path or result.path or result.metadata.get("file_path")
```

---

### 3.4 Symbol results 没有聚合到文件级

symbol-only 检索天然返回函数 / 类 / 方法级结果，但 benchmark 的 Top-k 是文件级。

如果返回结果是：

```text
src/indexer.py::build_index
src/indexer.py::parse_file
src/indexer.py::scan_repo
```

这三个 symbol 实际上都属于同一个文件。如果 evaluator 直接把它们当成 Top-3，就会浪费 Top-k，并造成文件级指标异常。

因此需要把 symbol result 聚合成 file-level result：

```text
多个 symbol 命中同一个 file
-> 聚合为一个 file result
-> file score 使用该文件下最高 symbol score 或加权得分
-> 保留 matched_symbols 用于 function hit 和 failure report
```

---

### 3.5 symbol_only 不应该 fallback 到 keyword

修复 symbol-only 时不能为了让分数变好而 fallback 到 keyword 检索。

错误做法：

```text
symbol_only 找不到
-> fallback keyword
```

这样会污染 baseline，导致 symbol_only 失去诊断意义。

正确做法：

```text
symbol_only 只使用 symbol index
但必须保证 symbol index 构建、检索、结果格式、文件聚合、evaluator 兼容都正常
```

symbol_only 的价值不是刷高分，而是诊断 AST symbol index 是否真的有效。

---

## 4. 问题二：Docker daemon 不可用并 fallback 到 subprocess

当前提示：

```text
Docker daemon is not available. Falling back to subprocess sandbox.
```

这说明 RepoMind 原本尝试使用 Docker sandbox，但当前运行环境没有可用的 Docker daemon，于是自动回退到了本地 subprocess sandbox。

这个行为本身不一定是错误。

如果只是运行 benchmark，subprocess fallback 通常可以接受。但如果项目想展示：

```text
RepoMind 支持 Docker 沙箱隔离执行 patch 验证和测试运行
```

那么需要把 sandbox 行为做得更清楚、更可配置。

---

## 5. Docker fallback 的问题

当前提示可能存在几个问题：

1. 用户不知道这是 warning 还是 error。
2. 用户不知道 benchmark 是否还可信。
3. 用户不知道如何强制使用 Docker。
4. 用户不知道如何显式禁用 Docker warning。
5. README 没有解释 `DockerSandbox` 和 `SubprocessSandbox` 的区别。
6. Windows 下 Docker Desktop 没启动时容易误以为项目坏了。

因此建议增加显式 sandbox 参数：

```bash
--sandbox auto
--sandbox docker
--sandbox subprocess
```

默认使用：

```bash
--sandbox auto
```

---

## 6. 推荐 sandbox 行为

### 6.1 auto 模式

```bash
uv run python eval/run_host_comparison.py --sandbox auto
```

行为：

```text
Docker 可用 -> 使用 DockerSandbox
Docker 不可用 -> warning，然后 fallback 到 SubprocessSandbox
```

这是默认模式，适合普通用户。

---

### 6.2 docker 模式

```bash
uv run python eval/run_host_comparison.py --sandbox docker
```

行为：

```text
Docker 可用 -> 使用 DockerSandbox
Docker 不可用 -> 直接报错退出
```

适合需要严格沙箱隔离的场景。

错误信息应该清楚说明：

```text
Docker sandbox requested, but Docker daemon is not available.
Please start Docker Desktop or use --sandbox subprocess.
```

---

### 6.3 subprocess 模式

```bash
uv run python eval/run_host_comparison.py --sandbox subprocess
```

行为：

```text
直接使用 SubprocessSandbox
不检测 Docker
不打印 Docker warning
```

适合本地开发、Windows 环境、快速 benchmark。

---

## 7. README 中应该如何解释

建议 README 添加：

````md
### Sandbox Modes

RepoMind supports three sandbox modes:

| Mode | Behavior |
|---|---|
| auto | Use Docker if available, otherwise fallback to subprocess |
| docker | Require Docker daemon and fail fast if unavailable |
| subprocess | Run locally without Docker isolation |

For local benchmark evaluation, `subprocess` mode is usually enough:

```bash
uv run python eval/run_host_comparison.py --compare-modes --sandbox subprocess
````

For isolated patch verification, start Docker Desktop and use:

```bash
uv run python eval/run_host_comparison.py --sandbox docker
```

On Windows, make sure Docker Desktop is running before using Docker sandbox.

````

## 8. 当前面试讲法

如果面试官问：

> 你这个 symbol_only 为什么是 0？

可以回答：

> 这个结果说明 symbol-only baseline 目前还有工程链路问题，不应该把它包装成正常结果。我的判断是 symbol index、symbol result schema、file-level aggregation 或 evaluator 兼容性中至少有一处没有打通。后续修复方向是先确保 AST symbol index 真的构建成功，再让 symbol result 映射回 file_path，并聚合到 file-level result，最后让 evaluator 能正确计算 Top-k File Hit 和 Function Hit。

如果面试官问：

> Docker daemon 不可用是不是项目有问题？

可以回答：

> 这不是 benchmark 本身失败，而是沙箱后端不可用。当前项目在 Docker 不可用时 fallback 到 subprocess sandbox，保证本地 benchmark 仍然能跑。后续我会把它改成显式 `--sandbox auto/docker/subprocess` 三种模式，让用户可以选择严格 Docker 隔离或者本地 subprocess 快速运行。

## 10. 最终结论

当前 RepoMind 的 full 模式已经有较好的展示价值：

```text
Top-1 File Hit: 76%
Top-3 File Hit: 90%
````

但 `symbol_only = 0` 和 Docker fallback 提示说明项目仍有工程细节需要补齐。

这两个问题的修复重点不是“提高表面分数”，而是让系统更加可信、可解释、可配置：

1. symbol_only 要证明 AST symbol index 是真实有效的。
2. sandbox 要让用户明确知道当前使用的是 Docker 隔离还是本地 subprocess。
3. README 要把实验模式、fallback 机制和当前限制讲清楚。

这样项目在面试中会更像一个认真迭代过的工程系统，而不是一个只追求跑分的 demo。
