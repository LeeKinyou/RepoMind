## 二、逐项修复详情

### H10. _safe_symbol_type 重复定义 — 违反 DRY

**根因**: v1 修复 M1 时，在 query_service.py 和 rca_service.py 中各复制了一份 `_safe_symbol_type` 函数。当时只关注"让它工作"，没有审视代码组织。

**修复**:
- `schemas.py`: 新增 `safe_symbol_type()` 公开函数
- `query_service.py` / `rca_service.py`: 删除本地定义，改为 `from repomind.models.schemas import safe_symbol_type`

**反思**: 修复 bug 时不能只看单个文件。当一个工具函数需要在多个模块使用时，应该立即放到公共位置（models/schemas.py 或 utils/），而不是"先复制一份让它跑起来"。

---

### H11. GraphStore.get_subgraph 传 str 而非 SymbolType

**根因**: `get_subgraph` 从 NetworkX 节点属性读取 `type` 字段，存的是 `str`（`"function"`），但 `SymbolInfo.type` 期望 `SymbolType` 枚举。Pydantic v2 的隐式转换在某些情况下能工作，但不保证稳定。

**修复**: 使用 `safe_symbol_type()` 显式转换。

**反思**: 数据流跨越存储层（NetworkX dict）和模型层（Pydantic model）时，边界处必须做类型转换。v1 修复 M1 时只在 query_service 和 rca_service 做了转换，漏掉了 graph_store 这条路径。

---

### H7. 签名提取 split(":")[0] 截断类型注解

**根因**: `def foo(x: str) -> int:` 中，第一个 `:` 在参数类型注解处。`split(":")[0]` 得到 `def foo(x`，丢失了完整签名。

**修复**: 用 tree-sitter 的 `body_node.start_byte` 精确切取函数定义部分（不含 body）：
```python
sig_bytes = pf.source[node.start_byte:body.start_byte]
signature = sig_bytes.decode("utf-8").rstrip().rstrip(":").strip()
```

**反思**: 字符串分割是脆弱的解析策略。tree-sitter 已经提供了精确的 AST 节点边界，应该利用它而不是自己做文本分割。v1 修复时看到了这个问题但用了"够用就行"的方案，没有彻底解决。

---

### H8. 返回类型注解解析对复杂类型出错

**根因**: `split("->")[1].split(":")[0]` 对 `dict[str, list[int]]:` 这样的复杂类型会截断错误。

**修复**: 改用正则 `->\s*([\w\[\],\s\.]+)\s*:` 精确匹配返回类型。

**反思**: 与 H7 同类问题 — 用字符串操作解析结构化文本。正确的做法是利用 tree-sitter AST 节点，或者至少用正则表达式。这次修 H7 时用 tree-sitter 解决了签名，但 `_strategy_explicit_hint` 仍然在做行级文本解析，说明修复时没有系统性地审视所有同类代码。

---

### H13. _extract_call 后不继续递归 — 嵌套调用丢失

**根因**: `_walk_node` 遇到 `call` 节点时调用 `_extract_call` 后直接跳过，不递归进入子节点。`foo(bar())` 中 `bar()` 的调用信息丢失。

**修复**: 在 `_extract_call` 后添加 `self._walk_node(child, ...)` 继续递归。

**反思**: v1 修复时添加了函数体递归（`_walk_node(body, ...)`），但只考虑了"进入函数体"，没有考虑"进入调用子树"。这是典型的"修了一个 bug，没检查相邻代码是否有同类问题"。

---

### C5. IndexService 传空字典给 resolve_callee

**根因**: `CallGraphBuilder.build()` 内部正确构建了 `_symbol_index`，但 `IndexService` 存储调用关系到 SQLite 时，自己调用 `SymbolResolver.resolve_callee(call, ..., {})`，传了空字典。

**修复**: 在 index_service.py 中构建 `symbol_index` 并传入。

**反思**: 这是典型的"两处代码做同一件事"问题。v1 修复 M2（统一 resolver）时，只统一了 graph_builder 的逻辑，没有检查 index_service 是否也在做同样的事。应该 grep 整个项目找所有调用点。

---

### M12. symbol_index 用短名作键 — 同名函数互相覆盖

**根因**: `self._symbol_index[sym.name] = sym.qualified_name` 用短名作键。`utils.parse` 和 `core.parse` 同名时，后解析的覆盖先解析的。

**修复**: 改为 `dict[str, list[str]]`，每个短名映射到所有匹配的 qualified_name 列表。resolver 优先选择与 caller 同模块的候选。

**反思**: 这是一个设计假设错误 — 假设短名全局唯一。在大型 Python 项目中，同名函数极为常见（如 `__init__`、`get`、`parse`）。设计数据结构时应该考虑最坏情况，而不是"大多数时候能工作"。

---

### M13. BM25 分词对代码符号效果差

**根因**: `text.lower().split()` 按空格分词。`get_user_info` 是一个 token，搜索 "user" 或 "info" 无法匹配。

**修复**: 新增 `_tokenize()` 方法，按 `[_.\s]+` 分割，`get_user_info` → `["get", "user", "info"]`。

**反思**: BM25 是为自然语言设计的，直接用于代码搜索需要适配。v1 实现 BM25 时直接用了 `.split()`，没有思考代码符号的分词特性。应该在设计阶段就考虑"代码搜索和文本搜索有什么不同"。

---

### M14. VisualizationService._node_shape 参数类型不匹配

**根因**: 方法签名 `_node_shape(self, sym_type: str)` 接受 `str`，但调用方传入 `SymbolType` 枚举。虽然 `SymbolType(str, Enum)` 的隐式比较能工作，但类型不安全。

**修复**: 改为 `_node_shape(self, sym_type: SymbolType)` 并用枚举值比较。

**反思**: Python 的 `str, Enum` 双继承让 str 和枚举可以混用，掩盖了类型不匹配问题。应该坚持使用枚举，不要依赖隐式行为。

---

### M15. CLI query 命令不显示 answer

**根因**: `query` 命令打印了结果列表和统计信息，但从未显示 `QueryResult.answer`。用户看不到 AI 生成的回答。

**修复**: 在结果列表前添加 `console.print(f"\n[bold]{result.answer}[/]\n")`。

**反思**: 这是功能遗漏。实现 CLI 时关注了"展示搜索结果"，忽略了"展示回答"。应该对照 API 返回值逐字段检查 UI 是否展示了所有有用信息。

---

### M16. duck_typing 推断匹配注释和字符串

**根因**: `_strategy_duck_typing` 对整个 source 做 `re.findall(pattern, source)`，会匹配到 `# obj.method(` 注释和 `"obj.method("` 字符串中的内容。

**修复**: 逐行处理，跳过 `#` 开头的注释行，移除行内 `#` 注释后再匹配。

**反思**: 正则匹配源码时必须考虑注释和字符串。v1 修复 M4（排序稳定性）时只关注了输出格式，没有检查输入质量。"垃圾进，垃圾出" — 如果匹配到了注释中的内容，排序再稳定也是错的。

---

### M18. RCA _frame_to_qname 与索引路径转换不一致

**根因**: `_frame_to_qname` 手动 `replace("/", ".").replace("\\", ".").replace(".py", "")`，但索引时用 `path_to_module()`。两处逻辑不同，导致 RCA 查找符号时匹配失败。

**修复**: 改用 `path_to_module()` 统一转换。

**反思**: v1 修复 H1（Windows 路径）时创建了 `path_utils.py` 并统一了 parser 和 index_service 的路径转换，但漏掉了 rca_service。应该用 grep 搜索所有 `.replace("/", ".")` 和 `.replace("\\", ".")` 的调用，确保全部替换。

---

### L8. VectorStore._get_table 异常捕获过宽

**根因**: `except Exception` 捕获所有异常后静默创建新表，掩盖了连接错误、权限问题等非"表不存在"的异常。

**修复**: 收窄为 `(ValueError, KeyError, OSError)`。

**反思**: "让它能跑"的心态导致了宽泛的异常捕获。应该明确"什么异常表示表不存在"，只捕获那几种。

---

### L9. _collect_files 中 fp.stat() TOCTOU 竞态

**根因**: `rglob("*.py")` 返回文件路径后，文件可能在 `stat()` 前被删除。

**修复**: `fp.stat()` 包裹 `try/except OSError`。

**反思**: 这是并发/竞态问题的典型模式。文件系统操作天然不稳定，所有 `stat()`/`open()` 调用都应该有防御性处理。

## 三、根因分析 — 为什么 v1 之后还有这么多问题

### 3.1 修复粒度问题："修一个点"而非"修一条线"

v1 修复时，每个问题都是独立处理的：
- 修了 M1（`_safe_symbol_type`），但只在 query_service 和 rca_service 各复制了一份，没有放到公共位置
- 修了 H1（`path_to_module`），但只替换了 parser 和 index_service，漏掉了 rca_service
- 修了 M2（SymbolResolver），但只统一了 graph_builder，漏掉了 index_service 的调用

**教训**: 修一个函数/方法时，应该 grep 整个项目找所有调用点和同类代码，确保全部同步修改。

### 3.2 设计假设过于乐观

- **symbol_index 用短名作键**: 假设函数名全局唯一，但 `__init__`、`get`、`parse` 在大项目中随处可见
- **BM25 用空格分词**: 假设代码符号和自然语言一样按空格分词，但 `get_user_info` 是一个整体
- **duck_typing 匹配全文**: 假设源码中只有"真正的代码"，但注释和字符串中也有 `name.method(` 模式

**教训**: 设计数据结构和算法时，应该问"最坏情况是什么"，而不是"大多数时候能工作吗"。

### 3.3 缺少端到端测试

v1 的 138 个测试覆盖了各模块的单元逻辑，但缺少跨模块的端到端测试：
- 没有测试"索引 → 查询 → 图扩展"完整流程
- 没有测试"索引 → RCA → 符号查找"完整流程
- 没有测试 BM25 搜索 `get_user_info` 时能否匹配 "user"

**教训**: 单元测试只能保证"每个零件合格"，不能保证"组装后能用"。关键路径必须有端到端测试。

### 3.4 类型系统利用不足

Python 的类型注解是可选的，导致多处类型不匹配被忽略：
- `GraphStore` 存 `str`，`SymbolInfo` 要 `SymbolType` — 没有在边界处做转换
- `_node_shape` 参数声明 `str`，实际传入 `SymbolType` — 隐式比较掩盖了问题
- `symbol_index` 类型从 `dict[str, str]` 改为 `dict[str, list[str]]`，但 resolver 的签名没同步更新

**教训**: 类型注解不是装饰品。应该用 mypy 检查，至少在 CI 中运行一次。

### 3.5 "能跑就行"的代价

v1 的目标是"从零到能跑"，追求速度而非质量。这导致：
- 多处用字符串分割代替结构化解析（H7, H8）
- 异常捕获过宽（L8）
- 没有考虑并发安全（L9）
- 函数复制而非提取公共模块（H10）

**教训**: "能跑就行"的技术债会以指数级累积。第一次就应该做对，否则第二次修复的成本是第一次的 3-5 倍。

## 四、改进措施 — 如何避免下一轮

### 4.1 修复时的 Checklist

每次修复一个 bug 时，必须完成以下步骤：

```
□ grep 整个项目，找所有同类代码
□ 检查所有调用点是否需要同步修改
□ 检查数据流的上下游是否受影响
□ 添加/更新测试覆盖修复
□ 运行 mypy 检查类型一致性
□ 运行 ruff 检查代码规范
```

### 4.2 设计阶段的审查清单

设计数据结构和算法时，必须回答：

```
□ 键/标识符是否可能重复？（如短名函数）
□ 输入数据是否可能包含噪声？（如注释、字符串）
□ 边界处是否做了类型转换？（如 str → enum）
□ 最坏情况下的时间/空间复杂度是什么？
□ 是否有并发安全问题？（如文件系统操作）
```

### 4.3 测试策略升级

```
当前: 138 个单元测试（模块级）
目标: 单元测试 + 集成测试 + 端到端测试

必须覆盖的端到端路径:
  1. 索引项目 → 查询符号 → 获取调用图 → 验证结果非空
  2. 索引项目 → RCA 分析 → 验证能找到符号
  3. 索引项目 → 搜索 "user" → 验证能找到 get_user_info
  4. 索引 → 清除 → 重新索引 → 验计数一致
```

### 4.4 静态分析集成

```bash
# 在 CI 中运行
uv run mypy src/repomind/ --strict     # 类型检查
uv run ruff check src/ tests/          # lint
uv run pytest --cov=src/repomind       # 测试覆盖率
```

### 4.5 代码审查时的关注点

```
1. 数据流完整性: 从存储层到模型层，每个边界是否有类型转换？
2. 错误处理精确性: except Exception → except SpecificError
3. DRY 原则: 是否有重复代码？是否应该提取公共函数？
4. 最坏情况假设: 键是否唯一？输入是否干净？并发是否安全？
5. 测试覆盖: 修复的 bug 是否有对应的回归测试？
```

## 六、结论

v2 报告的 14 个新问题，本质上是 v1 修复不彻底的后遗症。根本原因不是"能力不足"，而是"修复时缺乏系统性思维" — 只修了眼前的 bug，没有审视上下游、没有搜索同类代码、没有考虑最坏情况。

**一句话总结**: 修 bug 不是"改一行代码"，而是"改一条数据流"。每次修复都应该问：这条数据流上还有多少个同类问题？
