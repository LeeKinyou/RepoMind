# RepoMind 全面代码审查报告

## 2026-06-21 复审增量

本节记录在当前 CLI v3 工作树基础上进行的再次全仓库审查。现有测试全部通过，但关键真实工作流仍存在数据正确性问题。

### P1：重复索引不幂等，且不会清理删除文件

- 同一文件内容不变时，`upsert_file()` 返回已有 file id，但后续仍然重复插入 symbols、imports 和 type info。
- 实测同一项目连续索引两次后，symbol 数量从 1 增长为 2。
- 源文件删除后重新索引，旧 files、symbols 和 graph nodes 仍然存在。

建议：全量索引开始时建立当前文件集合，按文件事务替换相关记录，并删除不再存在的文件；增量模式必须根据 hash 明确跳过或重建。

### P1：调用图 caller 被错误归属到模块或类

`CallInfo` 只保存 `caller_class`，没有保存当前函数或方法。当前行为：

```text
def caller(): callee()
pkg.mod -> pkg.mod.callee
```

正确关系应为：

```text
pkg.mod.caller -> pkg.mod.callee
```

类方法调用同样被归到类节点。该问题会影响 `/callers`、`/callees`、图扩展和 RCA。

### P1：REPL 执行 `/index` 后 QueryService 保持旧状态

`IndexService` 与 `QueryService` 分别持有独立的 GraphStore。索引完成回调只更新项目注册表，没有重新加载 QueryService 的 graph，也没有使已构建的 BM25 缓存失效。

实测索引完成后：

```text
IndexService graph nodes: 3
existing QueryService graph nodes: 0
fresh QueryService graph nodes: 3
```

建议：索引成功后统一刷新查询服务，或让索引与查询服务共享同一个 storage context。

### P1：qualified name 包含绝对路径

`TreeSitterParser.parse_file()` 未接收 project root，直接将绝对路径传给 `path_to_module()`。Windows 下会生成类似：

```text
F:.repo.pkg.mod.symbol
```

这会导致符号 ID 与机器目录绑定，并使项目内导入名无法与索引符号稳定匹配。应统一使用相对于项目根目录的模块路径。

### P2：部分失败仍固定返回 `success=True`

文件解析、SQLite 存储、调用图构建或 graph 保存失败都只追加到 `errors`，最终 IndexResult 仍固定为成功。数据库逐条提交也可能留下半成品索引。

建议：定义 warning 与 fatal error 边界，并使用文件级或索引级事务。

### P2：VectorStore 与依赖声明不一致

`storage/vector_store.py` 仍然直接导入 `lancedb`，但 `pyproject.toml` 和 `uv.lock` 已移除该依赖。全新安装后 VectorStore 无法导入，而测试通过 `pytest.importorskip("lancedb")` 静默跳过。

建议：如果保留 VectorStore，则恢复明确依赖并将向量检索接入 HybridRetriever；否则删除未使用实现和对应产品文案。

### P2：显式类型推断忽略常见内置类型

`_strategy_explicit_hint()` 要求返回类型首字母大写，因此 `bool`、`str`、`list[str]`、`dict[str, int]` 都不会被识别。

建议：接受合法 Python annotation 表达式，不要用首字母大小写判断类型有效性。

### P2：SQLite 连接生命周期不完整

REPL 切换项目时直接创建新服务，没有关闭旧 SQLiteStore。覆盖率测试产生 29 个 `unclosed database` ResourceWarning，Windows 下还可能造成数据库文件锁。

建议：为服务增加 `close()`，项目切换和 REPL 退出时显式释放资源。

### 验证基线

- pytest：171 passed，1 skipped

- Ruff：2 个未使用 import
- `git diff --check`：文档中存在 3 处行尾空格

## 一、严重问题（必须立即修复）

### S1. `call_graph/graph_builder.py` — 导入边查找逻辑 Bug

**文件**: `src/repomind/core/call_graph/graph_builder.py` 第 30 行

```python
qname = self._resolve_import(imp)
if qname and qname in self._symbol_index:  # <-- BUG
```

`_resolve_import` 返回限定名（如 `"os.path.join"`），但 `_symbol_index` 的键是短名称（如 `"join"`）。`qname in self._symbol_index` 几乎永远为 `False`，导致**整个导入关系图几乎为空**。

**修复**: 提取短名称后查找，或检查 `qname` 是否存在于 values 中：

```python
short_name = qname.rsplit(".", 1)[-1] if "." in qname else qname
if short_name in self._symbol_index:
    # 进一步验证 qname 确实在 self._symbol_index[short_name] 列表中
```

### S2. `storage/graph_store.py` — pickle.loads() 任意代码执行风险

**文件**: `src/repomind/storage/graph_store.py` 第 95-110 行

```python
# save
data = pickle.dumps(self.graph)
sig = hmac_mod.new(b"repomind", data, hashlib.sha256).hexdigest()

# load
self.graph = pickle.loads(data)
```

HMAC 密钥硬编码为 `b"repomind"`。任何拥有源代码的人都可以构造恶意 pickle 数据并通过签名验证，导致**任意代码执行**。

**修复**: 替换为 `networkx.node_link_data()` / `node_link_graph()` 配合 JSON 序列化：

```python
def save(self, path: str) -> None:
    data = nx.node_link_data(self.graph)
    with open(path, "w") as f:
        json.dump(data, f)

def load(self, path: str) -> None:
    with open(path) as f:
        data = json.load(f)
    self.graph = nx.node_link_graph(data)
```

### S3. `storage/vector_store.py` — delete() SQL 注入风险

**文件**: `src/repomind/storage/vector_store.py` 第 63-67 行

```python
def delete(self, ids: list[str]) -> None:
    table = self._get_table()
    sanitized = [i.replace("'", "''") for i in ids]
    id_list = ", ".join(f"'{s}'" for s in sanitized)
    table.delete(f"id IN ({id_list})")
```

手动 SQL 转义仅替换单引号，容易遗漏攻击向量（反斜杠转义、Unicode 绕过等）。

**修复**: 检查 LanceDB 是否支持参数化 delete，或使用 UUID 格式校验限制输入。

### S4. `storage/sqlite_store.py` — 缺少上下文管理器协议

**文件**: `src/repomind/storage/sqlite_store.py` 第 109-148 行

类有 `close()` 方法但没有实现 `__enter__`/`__exit__`，无法配合 `with` 语句使用。如果使用者忘记调用 `close()`，数据库连接将泄漏。

```python
def __enter__(self):

```

### S5. `core/parser/tree_sitter_parser.py` — 链式调用解析丢失信息

**文件**: `src/repomind/core/parser/tree_sitter_parser.py` 第 260-267 行

```python
def _get_attribute_parts(self, node: Node) -> list[str]:
    parts = []
    for child in node.children:
        if child.type == "identifier":
            parts.append(self._node_text(child))
        elif child.type == "attribute":
            parts.extend(self._get_attribute_parts(child))
    return parts
```

在 `a.b.c().d` 这样的链式调用中，`call` 节点被忽略。结果 `a.b.c().d` 被解析为 `["a", "b", "d"]` 而丢失 `"c"`。

### S8. 核心 AI 推理层（LiteLLM）完全缺失

**文件**: `src/repomind/services/query_service.py` 和 `src/repomind/services/rca_service.py`

在当前代码中，除了 `config.py` 中定义了 Pydantic 模型的配置项外，系统层和服务层完全没有引入 `litellm` 并调用大语言模型进行 AI 推理或生成解答。所有查询只是进行关键词和图谱检索，RCA 排错分析仅是通过正则表达式解析堆栈帧和符号名并以字符串模板拼接来作为 explanation 输出。

**影响**: RCA 分析仅能给出 proximate cause（直接报错符号），无法利用大语言模型提供高置信度的根因分析与代码逻辑漏洞定位，也无法生成具体的代码修复建议。自然语言查询只能查找匹配的符号列表，无法提供针对性回答。

**修复**: 接入 `litellm`。在 `QueryService` 中将混合检索与图拓扑扩展获取的上下文输入 LLM，生成高质量的解答；在 `RCAService` 中将堆栈帧相关的源码片段与错误信息组装为 prompt 输入 LLM，生成深入的根因诊断与修复方案。

---

### S9. 核心沙箱执行器（Sandbox Executor）完全缺失

**文件**: `src/repomind/sandbox/executor.py`

在项目设计与文档中宣称使用 Docker 沙箱进行安全代码执行（禁用网络、只读挂载、资源限制），以确保执行本地或外部命令时不带任意代码执行风险。但目前 `src/repomind/sandbox/executor.py` 是一个几乎为空的文件，没有任何沙箱管理或执行逻辑。

**影响**: 无法在安全的隔离沙箱中执行分析或排错检测代码，系统对未知代码的诊断只能停留在静态分析层面，或者直接在本地操作系统中执行，从而带来严重的任意代码执行等安全风险。

**修复**: 实现 `sandbox/executor.py` 中基于子进程限制或 Docker 容器的安全隔离代码执行逻辑。

### H1. `call_graph/graph_builder.py` — 父类解析盲目取第一个候选

### H2. `call_graph/resolver.py` — O(N) 线性扫描性能问题

**文件**: 第 52-55 行

```python
for name, qnames in symbol_index.items():
    for qname in qnames:
        if qname.endswith(f".{target}"):
            return qname
```

每次调用解析遍历整个符号索引。对于大型代码库（数万符号），总复杂度为 O(C * N)。

**修复**: 构建反向索引 `suffix_index: dict[str, list[str]]`。

### H3. `call_graph/` — 过度使用裸 dict

`ParsedFile` 的 `imports`、`calls`、`classes` 字段以及 `graph_builder.py` 和 `resolver.py` 中的所有参数都使用 `dict`，导致：
- 无法静态检查键名正确性（typo 不会被发现）
- IDE 无法提供自动补全
- 无法保证所有必要字段都存在

**修复**: 定义结构化类型：

```python
class CallInfo(TypedDict):
    caller_class: str | None
    target: str
    call_type: str  # "direct" | "self" | "method"
    line_number: int

class ImportInfo(TypedDict):
    module_path: str
    imported_name: str | None
    alias: str | None
    is_relative: bool
    relative_level: int
    line_number: int
```

## 六、CLI 用户体验与交互设计问题（对比 Claude Code / OpenCode）

当前项目的 CLI 在交互设计与用户体验上较为基础，无法给用户明确的使用引导，具体交互模式对开发人员来说有些粗糙，不够直观。我们可以对比 Claude Code 和 OpenCode 的先进 CLI 设计，找出以下待提升的问题与缺陷，并给出具体的重构设计方案：

### UX1. 缺少交互式 Onboarding 引导向导
**现状**: 当用户在终端直接输入 `repomind` 启动交互模式时，如果当前项目尚未进行索引扫描，系统只会在启动 Banner 中以黄色打印 `Index: Not indexed`。随后直接呈现一个空白 Prompt，用户不知道该做什么、可以做什么、第一步应该做什么。
*   **Claude Code 风格建议**: 启动时检查索引数据库是否存在。如果未索引，立即主动展示 Onboarding 交互向导，提示用户是否需要立即对当前目录建立索引，提供一键引导：
    ```
    [!] RepoMind: No repository index found in the current directory.
    [*] Would you like to run /index to build the index now? [y/N]:
    ```

### UX2. 终端 Prompt 缺乏上下文环境感知
**现状**: 目前交互模式下的 Prompt 只是一个单纯 of `> `（加绿色高亮），这在较长的终端窗口中极其容易丢失视觉焦点，且完全无法反馈当前的运行上下文。
*   **Claude Code / OpenCode 风格建议**: 升级为富文本动态 Prompt，实时反馈当前项目的名称、Git 分支、以及当前索引的状态。例如：
    *   未索引状态: `repomind [RepoMind (unindexed)] > ` （使用黄色或灰色标注，提醒用户索引状态）
    *   已索引状态: `repomind [RepoMind|main] > ` （使用绿色高亮）
    这样可以帮助用户感知当前处于哪一个工程目录与分支下。

### UX3. 搜索/查询结果缺少交互式数字选择器与动作触发
**现状**: 当运行 `/query <question>` 或输入自然语言进行检索时，系统会输出一个格式化表格并展示匹配的符号（10个结果）。然而，用户想要阅读其中某个具体符号的源码时，必须手动输入或复制超长的完全限定名（如 `/show repomind.core.parser.tree_sitter_parser.Parser`），不仅操作繁琐，而且容易拼写错误。
*   **Claude Code / OpenCode 风格建议**: 采用交互式数字选择机制。搜索出的结果以带有编号的面板形式展现（例如 `1.`、`2.`...），搜索结束后，Prompt 转换为交互选择提示：
    ```
    [i] 输入序号 [1-10] 查看对应符号详情，或按 Enter/Q 返回主界面:
    ```
    用户只需键入数字 `1` 即可直接触发 `/show` 命令查看该符号源码，配合动作智能分流。

### UX4. 缺乏后续操作智能推荐（Next Steps Suggestions）
**现状**: 目前 CLI 的各个命令之间是割裂 of。用户运行完 `/index` 看到 "Index complete" 后，只看到下一个 Prompt `> `，没有智能提示来引导后续动作。同样，在进行 `/query` 或 `/rca` 之后也是如此。
*   **Claude Code / OpenCode 风格建议**: 在每个重点命令执行成功后的输出最下方，提供一行 context-aware 的智能下一步行动指南。例如：
    *   `/index` 完成后: `[*] 建议：直接输入自然语言提问（例如 "Parser 是在哪里被调用的？"）或运行 /query 搜索符号。`
    *   `/query` 完成后: `[*] 建议：输入对应的数字序号查看源码，或运行 /graph <名称> 查看其调用链路。`
    *   `/rca` 完成后: `[*] 建议：运行 /show <疑似符号> 查看可能存在缺陷的代码段。`

### UX5. 缺失 Tab 键自动补全与模糊纠错机制
**现状**: 在交互终端中，目前不支持 Tab 键自动补全（例如输入 `/h` 无法自动补全为 `/help`，输入类名或函数名时也无法补全），当用户手误输入了无效指令（如 `/idx`）时，系统仅冷冰冰地报错 `Unknown command`。
*   **Claude Code / OpenCode 风格建议**:
    *   **Tab 补全**: 集成 Python 的 `prompt_toolkit`，支持命令的 Tab 补全，甚至在输入参数时对已索引的符号进行联想补全。
    *   **模糊纠错**: 引入编辑距离（Levenshtein distance）计算。当用户输入 `/idx` 时，报错提升为：
        ```
        [Error] 未知指令 "/idx"。您是不是想运行 /index ？
        ```

### UX6. `/rca` 异常分析接口缺乏交互式粘贴机制
**现状**: 目前的 `/rca` 必须要求通过参数传递文件路径（如 `/rca --trace error.log`）。如果用户直接输入 `/rca` 且无参数，系统容易报错或让用户不知道该怎么输入 Trace。
*   **Claude Code 风格建议**: 支持交互式剪贴板 Trace 输入。当用户未指定文件直接输入 `/rca` 时，启动多行录入模式：
    ```
    [RCA] 请在下方粘贴您的 Traceback 堆栈信息 (在新行输入 Ctrl+D 或连续按两次 Enter 提交分析)：
    ------------------------------------------------------------------------
    [等待用户输入...]
    ```
    录入完成后，后台直接解析输入内容并生成诊断报告，免去了用户必须先将错误日志存盘的步骤。

### UX7. 代码片段查看缺少分页与折叠保护（Page View & Folding）
**现状**: 使用 `/show <symbol>` 查看符号详情时，如果符号是一个上百行的 God Class 或 God Method，终端会一次性全部灌入大段源码，导致历史记录被刷掉，用户无法看清前文。
*   **Claude Code / OpenCode 风格建议**: 引入代码分页与自动折叠控制：
    *   默认限制最多只打印 40 行，若代码超出此行数，自动折叠并提供分页提示：
        ```
        [第 1-40 行，共 180 行] 按 Enter 键继续查看，或输入 q 退出阅读:
        ```
    *   或者直接使用 Python 的内建 `pydoc` 模块所使用的 `pager`（如 `less` 模式）来流畅浏览大型代码段。

### UX8. 缺少全局交互式主菜单与直观命令导览（Interactive Command Menu Dashboard）
**现状**: 当直接通过 `repomind` 启动交互模式时，缺乏一个总览性的面板告诉用户系统能干什么、该干什么，也没有命令导航，全靠用户自己看 `/help`。
*   **Claude Code / OpenCode 风格建议**: 启动时除了 Banner，展示一个结构化的“主任务导航栏”（Task Navigation Menu），用简洁且美观的面板列出核心功能，配合当前状态标识，例如：
    ```
    ┌─────────────────────────── RepoMind Quick Actions ───────────────────────────┐
    │  [1] /index    - 索引当前代码仓库 (当前状态: 已索引 30 文件)                 │
    │  [2] /query    - 精确检索代码符号 / 自然语言提问                             │
    │  [3] /graph    - 探索符号的调用关系图谱 (可视化 Mermaid 输出)                │
    │  [4] /rca      - 输入异常 Trace 进行大模型根因诊断                           │
    │  [5] /stats    - 查看索引符号统计数据                                        │
    │  [6] /help     - 显示命令帮助手册                                            │
    └──────────────────────────────────────────────────────────────────────────────┘
    ```
    用户可以直接键入序号 `1`、`2` 等或者输入对应的命令，降低用户使用记忆负担。

### UX9. 缺乏交互执行时的多步骤动效反馈（Multi-step Live Feedback & Progress Dashboard）
**现状**: 目前诸如 `/index` 等长耗时任务，只有一个简单的 `Indexing项目...` 静态 spinner。用户看不到具体在扫描什么文件、卡在了哪个步骤，缺乏细节掌控感。
*   **Claude Code / OpenCode 风格建议**: 引入分步骤的渐进式任务看板，动态更新步骤状态和实时进度。例如：
    ```
    [*] Step 1/4: Scanning workspace files... -- Done (Found 42 Python files)
    [*] Step 2/4: Running AST parser & progressive type inference...
      [======================>                 ] 55% (23/42) - parser.py
    [-] Step 3/4: Building call graph relations... -- Waiting
    [-] Step 4/4: Persisting data structures... -- Waiting
    ```
    结合 Rich 的多任务 Progress 栏，能够提供极其优秀的视觉反馈，给用户极致专业的感觉。
