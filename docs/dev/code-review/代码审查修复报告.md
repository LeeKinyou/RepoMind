## 二、逐项修复详情

### Phase 1: 核心功能可用

#### C1. GraphStore 持久化 — 图扩展完全失效

**根因**: `GraphStore` 设计为纯内存结构，但架构文档要求跨服务共享图数据。`IndexService` 构建的图在索引完成后随进程消亡，`QueryService` 和 `RCAService` 各自创建空图。

**修复**:
- `graph_store.py`: 新增 `save(path)` / `load(path)` 方法，使用 pickle 序列化 NetworkX 图
- `index_service.py`: 索引完成后调用 `self.graph.save(str(self.index_dir / "graph.pkl"))`
- `query_service.py` / `rca_service.py`: `__init__` 中检查 `graph.pkl` 是否存在，存在则加载

**修复代码量**: 4 个文件，约 20 行新增

#### C4. RRF 融合公式错误

**根因**: 用 `int(1/score)` 近似 rank，但 BM25 分数不受 [0,1] 约束。score=10 时 rank=0，score=0.001 时 rank=1000，不同 source 的 score 量纲不同无法直接比较。

**修复**: 重写 `_rrf_fuse` — 先按 source 分组排序计算真实排名，再用 `1/(k + rank)` 公式融合。

**修复代码量**: 1 个文件，约 35 行替换

#### H1. Windows 路径分隔符

**根因**: 多处路径转换用 `.replace("/", ".")` 但未处理 `\`。Windows 上 `Path("C:\Users\foo.py")` 的字符串表示包含反斜杠。

**修复**: 创建 `utils/path_utils.py` 提供 `path_to_module()` 函数，使用 `PurePosixPath` 统一转为 POSIX 风格。4 个文件中的内联路径转换全部替换为调用此函数。

**修复代码量**: 新建 1 文件 + 修改 3 文件

#### M5. BM25 状态不重置

**根因**: `build()` 方法将 `doc_freqs` 和 `df` 追加而非替换。重新索引时旧数据累积。

**修复**: 在 `build()` 开头添加 `self.doc_freqs = []` 和 `self.df = Counter()`。

**修复代码量**: 2 行

### Phase 2: 安全与稳定性

#### C2. VectorStore SQL 注入

**根因**: `delete()` 方法用 f-string 直接拼接用户 ID，未转义单引号。

**修复**: 拼接前对每个 ID 执行 `i.replace("'", "''")` 转义。

**修复代码量**: 2 行

#### C3. config 整数解析崩溃

**根因**: `int()` 对空字符串或非数字字符抛出未处理的 `ValueError`。

**修复**: 新增 `_safe_int(value, default)` 辅助函数，catch `ValueError`/`TypeError`。

**修复代码量**: 约 8 行

#### M6. 增量索引数据残留

**根因**: `upsert_file` 在 hash 变化时只更新文件元数据，不清理旧的 symbols/imports/calls。`calls` 表没有 `ON DELETE CASCADE` 到 `files`。

**修复**: 在 UPDATE 前，按依赖顺序 DELETE 旧的 calls → inherits → type_info → symbols → imports，全部在同一事务中。

**修复代码量**: 6 条 SQL 语句

### Phase 3: 功能完善

#### H3. TypeInferenceEngine 未集成

**根因**: `IndexService.__init__` 创建了 `TypeInferenceEngine` 实例，但索引循环中从未调用。流水线缺少 `parse → infer types → store` 中间步骤。

**修复**:
- `sqlite_store.py`: 新增 `insert_type_info()` 方法
- `index_service.py`: 在 `insert_symbol` 后调用 `type_engine.infer_symbol()` 并存储结果

**修复代码量**: 约 15 行

#### H4. Parser 忽略 import 别名

**根因**: `_extract_import` 只提取 `dotted_name` 节点，未检查后续的 `as` 关键字。

**修复**: 重写为索引遍历，遇到 `dotted_name` 后检查下一个 token 是否为 `as`，是则提取别名。

**修复代码量**: 约 30 行替换

#### H5. 向量维度硬编码

**根因**: `_make_schema` 中 `pa.list_(pa.float32(), 1536)` 写死，不支持其他嵌入模型。

**修复**: `VectorStore.__init__` 增加 `embedding_dim` 参数，`_make_schema` 使用 `self.embedding_dim`。

**修复代码量**: 3 行

#### M2. 符号解析逻辑重复

**根因**: `IndexService._resolve_caller_qname` 与 `CallGraphBuilder._resolve_caller` 功能相同但独立实现，修改一边另一边不同步。

**修复**: 创建 `core/call_graph/resolver.py` 提供 `SymbolResolver` 类，两边统一调用。

**修复代码量**: 新建 1 文件 + 修改 2 文件（删除重复方法）

### Phase 4: 质量提升

#### H2. SQLite 连接复用

**根因**: 每个方法调用都 `sqlite3.connect()` → 设置 PRAGMA → `conn.close()`，索引 1000 符号 = 1000+ 次连接开关。

**修复**: 添加 `_get_conn()` 懒加载 + `close()` 方法，`_connect()` 复用缓存连接。

**修复代码量**: 约 15 行

#### M1. SymbolType 类型不安全

**根因**: SQLite 返回 `str`，`SymbolInfo.type` 期望 `SymbolType` 枚举。非法值导致 `ValidationError`。

**修复**: 新增 `_safe_symbol_type()` 函数，catch `ValueError` 回退到 `FUNCTION`。

**修复代码量**: 每个服务文件约 8 行

#### M3. docstring 提取错误

**根因**: `.strip("'\"")` 贪婪剥离所有引号字符，docstring 内容本身以引号开头/结尾时被误裁。

**修复**: 显式匹配 `"""` / `'''` / `"` / `'` 四种前缀后缀。

**修复代码量**: 5 行替换

#### M4. DuckType 排序不稳

**根因**: `set(matches)` 迭代顺序不确定，同一份代码两次运行可能产生不同结果。

**修复**: `sorted(set(matches))`。

**修复代码量**: 1 行

#### M7. LIKE 通配符未转义

**根因**: SQLite `LIKE` 将 `%` 和 `_` 视为通配符，用户搜索 `test_%` 时匹配任意字符。

**修复**: 转义 `\` → `\\`、`%` → `\%`、`_` → `\_`，SQL 中添加 `ESCAPE '\'`。

**修复代码量**: 3 行

#### L1-L3. 代码卫生

- **L1**: 5 个方法内部 `import re` 移至文件顶部
- **L2**: `metadata: dict = {}` 改为 `dict[str, Any] = Field(default_factory=dict)`
- **L3**: `clear()` 使用 `frozenset` 白名单校验表名

## 三、根因分析 — 为什么会出现这些问题

### 3.1 问题分类

将 19 个问题按根因归类：

| 根因类别 | 问题数 | 具体问题 |
|---------|--------|---------|
| **跨模块状态共享缺失** | 2 | C1 (GraphStore), M2 (SymbolResolver) |
| **算法/公式实现错误** | 2 | C4 (RRF), M5 (BM25 状态) |
| **输入验证不足** | 4 | C2 (SQL 注入), C3 (int 解析), M7 (LIKE 通配符), M1 (枚举转换) |
| **平台兼容性未考虑** | 1 | H1 (Windows 路径) |
| **功能集成遗漏** | 2 | H3 (TypeInference), H4 (import alias) |
| **硬编码/配置缺失** | 1 | H5 (向量维度) |
| **状态管理缺陷** | 2 | M5 (BM25 累积), M6 (增量残留) |
| **字符串处理不严谨** | 2 | M3 (docstring), M4 (DuckType 排序) |
| **性能设计粗糙** | 1 | H2 (连接复用) |
| **代码规范不一致** | 3 | L1-L3 |

### 3.2 深层原因

#### 原因 1: "能跑就行" 的实现心态

**表现**: C1、H3、H4 都是"写了个壳但没接通"的问题。`TypeInferenceEngine` 创建了但没调用，`GraphStore` 建了但没持久化，`import alias` 知道存在但没提取。

**本质**: 初始实现时优先追求"代码能跑通"，而非"功能完整正确"。每个模块单独看都是对的，但模块之间的数据流没有打通。

**反思**:
> 代码不是写给自己看的，是写给数据流看的。每写一个模块，都要问："这个模块的输出，谁来消费？消费方能正确拿到数据吗？"

#### 原因 2: 缺乏端到端验证

**表现**: C4 (RRF 公式)、M4 (DuckType 排序)、M3 (docstring) 都是逻辑错误，单元测试就能发现，但初始实现时没有写测试。

**本质**: 从文档到代码的转化过程中，缺少"写完就验证"的环节。代码写完后只做了导入检查，没有用真实数据跑一遍完整流程。

**反思**:
> 每个算法实现后，至少用一个手工 case 验证：输入已知，输出可算，对比预期。不能只靠"看起来对"。

#### 原因 3: 安全意识不足

**表现**: C2 (SQL 注入)、C3 (未处理异常)、M7 (LIKE 通配符) 都是"用户输入未做处理"的典型安全问题。

**本质**: 实现时只考虑了"正常输入"的情况，没有考虑"异常输入"或"恶意输入"。

**反思**:
> 凡是拼接用户数据到 SQL/Shell/Template 的地方，都是潜在注入点。写代码时要自问："如果这里传入 `'; DROP TABLE--` 会怎样？"

#### 原因 4: 跨平台假设

**表现**: H1 (Windows 路径) — 所有路径转换都假设 `/` 分隔符。

**本质**: 在 Unix 上开发时自然假设 POSIX 路径，但项目实际运行在 Windows 上。

**反思**:
> 涉及文件路径的代码，永远用 `pathlib` 而非字符串操作。`Path.as_posix()` 或 `PurePosixPath` 是跨平台的安全选择。

#### 原因 5: 重复代码未及时重构

**表现**: M2 (IndexService 和 CallGraphBuilder 各自实现了一遍符号解析)。

**本质**: 两个模块独立开发时各自实现了相同逻辑，没有在发现重复时立即提取公共代码。

**反思**:
> 当你发现自己在写"好像在哪见过"的代码时，停下来搜索一下。重复是 bug 的温床 — 修了一处忘了另一处。

#### 原因 6: 状态管理缺乏显式设计

**表现**: M5 (BM25 累积)、M6 (增量残留) — 都是"重入时旧状态未清理"。

**本质**: 方法设计时只考虑了"首次调用"的路径，没有考虑"重复调用"的场景。

**反思**:
> 写任何有状态的方法时，先问自己："如果这个方法被调用两次，第一次的状态会影响第二次吗？" 如果会，要么在开头重置，要么设计为幂等。

### 4.4 具体检查清单（供后续开发使用）

每次写完一个模块后，对照以下清单自查：

```
□ 跨模块共享: 这个模块的输出，其他模块能拿到吗？需要持久化吗？
□ 输入验证: 所有外部输入都做了验证/转义吗？
□ 状态管理: 方法被调用两次会出问题吗？需要重置吗？
□ 平台兼容: 路径操作用了 pathlib 吗？字符串 replace 考虑了 \ 吗？
□ SQL 安全: 有字符串拼接到 SQL 吗？转义了吗？
□ 类型安全: 枚举类型是从字符串转换的吗？有回退值吗？
□ 算法验证: 用一个手工 case 验证过输出吗？
□ 重复代码: 有没有其他地方实现了相同逻辑？
□ 魔法数字: 有硬编码的常量吗？应该参数化吗？
□ 导入规范: import 在文件顶部吗？有未使用的导入吗？
```

## 六、结论

本次代码审查发现的 19 个问题，根因集中在以下几点：

1. **模块间集成不充分** — 各模块独立实现但未打通数据流
2. **异常路径未覆盖** — 只考虑了正常输入和首次调用
3. **安全意识薄弱** — SQL 拼接、输入验证缺失
4. **平台假设单一** — 未考虑 Windows 环境
5. **缺少端到端验证** — 代码写完没有用真实数据跑完整流程

**核心教训**: 代码的正确性不在于单个函数的逻辑，而在于数据在模块间流转的完整性。写完一个模块后，最重要的检查不是"这个函数对不对"，而是"这个函数的输出能被下游正确消费吗"。

### 8.2 审查 vs 测试的互补关系

```
审查擅长发现:                      测试擅长发现:
├── 架构设计缺陷 (C1, M2)         ├── 边界条件 (H6 空字符串)
├── 安全漏洞 (C2, C6)             ├── 具体输入输出 (H7 签名截断)
├── 性能问题 (H2)                 ├── 模块间接口不一致 (C5 参数值)
└── 代码规范 (L1-L3)              └── 实际行为 vs 预期 (H8 复杂类型)
```

**关键发现**: 审查是"看代码找问题"，测试是"跑代码找问题"。两者互补，缺一不可。

### 8.3 TDD 暴露的深层模式

#### 模式 1: "接口对但参数错"

**表现**: `SymbolResolver.resolve_callee(call, caller_class, symbol_index)` 接口设计正确，但 `IndexService` 传入了 `{}` 空字典。

**本质**: 函数签名正确 ≠ 调用正确。审查只看签名，测试追踪数据流。

**教训**:
> 为每个跨模块调用写一个测试，验证传入的参数值是否符合预期，而非只验证参数类型。

#### 模式 2: "修了但没验"

**表现**: 第一轮修复 C6 时，审查报告提出了 HMAC 方案，但实际代码中 `save()`/`load()` 仍是裸 pickle。

**本质**: 修复方案提出 ≠ 修复方案落地。没有测试验证，修复就是空话。

**教训**:
> 每个修复必须对应一个失败的测试。测试变绿 = 修复完成。没有测试的修复 = 没有修复。

#### 模式 3: "字符串处理的隐含假设"

**表现**: `split(":")[0]` 假设冒号只出现在函数体开始处，`split("->")` 假设箭头只出现一次。

**本质**: 字符串处理函数对输入格式有隐含假设，这些假设在简单测试用例中成立，但在复杂用例中崩溃。

**教训**:
> 字符串处理函数必须用 3 种输入测试：(1) 最简单情况 (2) 含特殊字符 (3) 边界情况。对于签名/类型等结构化文本，优先用 AST 节点而非字符串分割。

#### 模式 4: "递归深度不够"

**表现**: `_walk_node` 递归遍历 AST，但 `_extract_function` 只提取了函数签名，没有递归进入函数体。

**本质**: 递归函数的"终止条件"设置不当 — 在应该继续递归的地方提前返回了。

**教训**:
> 写递归函数时，画一棵具体的 AST 树，手动走一遍每个节点的处理逻辑，确认每个分支都有递归或终止。

#### 模式 5: "1-based vs 0-based 混淆"

**表现**: `_strategy_self_inference` 用 `line`（1-based）做 `lines[line-1]`（0-based 索引），但 `range(line-1, ...)` 的起始位置算错了。

**本质**: 行号在不同上下文中含义不同 — tree-sitter 用 0-based，Python traceback 用 1-based，代码中混用了。

**教训**:
> 凡是涉及行号的函数，在 docstring 中明确标注 `line: 1-based line number` 或 `line: 0-based line index`。在函数入口处统一转换一次。
