# RepoMind 动态索引一致性与时效性方案

## 背景

RepoMind 的核心价值是为代码仓库诊断生成可复核的 Evidence Report。真实智能体工作流中，代码会被连续修改、测试、再次诊断。如果索引仍按“静态仓库”假设运行，就会出现旧索引解释新代码、旧调用图引用已删除符号、诊断报告无法判断是否过期等问题。

因此，动态一致性的核心目标不是追求“文件保存瞬间数据库物理强一致”，而是保证：

1. 每次诊断都基于明确的索引快照。
2. 查询前可以判断当前工作区是否仍匹配索引。
3. 代码变化后可以通过增量刷新恢复一致。
4. Evidence Report 后续可以被校验为 current 或 stale。

## 一致性语义

RepoMind 采用 snapshot-oriented consistency：

```text
query arrives
  -> check workspace freshness
  -> if stale: refresh incrementally or return stale
  -> run retrieval / graph expansion on one index_version
  -> bind report to index_version and file hashes
```

关键概念：

- `index_version`：每次成功索引后推进的快照版本。
- file manifest：由 SQLite `files` 表承担，记录 `path + hash + size + line_count`。
- freshness：当前磁盘文件 hash 与索引 manifest 的比较结果。
- stale report：当参与证据的文件 hash 变化或被删除时，旧报告不能继续被当作当前事实。

## 方案对比

### 方案 A：Watchdog Real-time Sync

文件监视器适合提升体验，但不能作为一致性的唯一来源。

优点：

- 能及时发现智能体保存文件、生成代码、删除文件等事件。
- 可以后台提前触发增量索引，减少下一次诊断等待。
- 对连续修改和多轮诊断体验明显更好。

限制：

- 文件事件可能重复、合并或丢失。
- 编辑器保存可能包含临时文件、rename、批量写入。
- 捕捉到事件不代表索引已经完成。
- 查询如果发生在后台刷新中，需要 snapshot isolation，否则可能读到半更新状态。

定位：

Watchdog 是 dirty-file discovery layer，不是 consistency layer。

### 方案 B：Incremental AST Diffing

函数级 AST diff 是性能优化，不是第一阶段的一致性底座。

优点：

- 可以减少超大文件中小改动带来的重复 symbol/embedding 重算。
- 适合长期驻留服务和高频编辑场景。

限制：

- import、decorator、class base、module constant、type alias 都可能影响多个函数。
- 函数 rename、移动、嵌套函数、同名方法会带来 symbol identity 难题。
- 调用图边依赖作用域、类型推断和解析策略，不能只看被修改函数体。
- 实现复杂度高，一旦 diff 错误会直接破坏诊断可信度。

定位：

AST diff 是后续 symbol-level optimization。第一阶段更稳妥的做法是文件级重建，并通过 symbol/chunk hash 复用 embedding。

### 方案 C：Git-diff Indexing

Git diff 是高收益的变更发现器，但覆盖不完整。

优点：

- 与智能体修改代码的 workflow 高度契合。
- 大仓库中能显著减少文件收集和 hash 扫描成本。
- 实现复杂度低，适合作为优先优化项。

限制：

- 只适用于 git repo。
- 默认 `git diff --name-only` 不覆盖 untracked 文件，需要配合 `git status --porcelain`。
- ignored 文件、生成文件、非 git 管理目录可能被漏掉。
- 分支切换、merge、reset 后仍需 manifest hash 校验。

定位：

Git diff 是 fast dirty-file detector，不能替代 manifest hash。

## 最终分层架构

```text
Change Detection Layer
  - manifest hash scan
  - git status / git diff fast path
  - watchdog dirty queue

Index Refresh Layer
  - file-level incremental rebuild
  - deleted-file cascade cleanup
  - per-symbol/chunk hash reuse
  - transactional index_version commit

Query Consistency Layer
  - query 前 freshness gate
  - stale 自动刷新或明确返回
  - retrieval / graph expansion 使用同一个 snapshot

Evidence Layer
  - report 绑定 index_version
  - report 绑定 file hashes
  - validate_evidence 判断 current/stale
```

## 推荐落地顺序

1. Manifest freshness check：用当前文件 hash 对比索引中的 file manifest。
2. Index version：成功索引后推进 `index_version`，报告和查询绑定该版本。
3. Incremental refresh：stale 时自动执行增量索引，新增、修改、删除文件都必须幂等。
4. Query/RCA freshness gate：诊断前统一调用 freshness check。
5. Git diff fast path：优先过滤 modified/untracked/deleted 文件，再用 hash 校验兜底。
6. Watchdog dirty queue：后台预刷新，查询前仍必须校验。
7. Symbol/chunk hash reuse：文件级重建时复用未变化 symbol 的 embedding。
8. Function-level AST diff：在正确性稳定后再做细粒度性能优化。

## 当前实现边界

本阶段已实现底座能力：

- `IndexService.check_freshness(path)`：返回 current/stale，并列出 changed/new/deleted files。
- `IndexService.refresh_if_stale(path)`：仅当 stale 时执行 incremental refresh。
- `IndexService.get_index_version()`：读取当前索引快照版本。
- 成功执行 `index_directory()` 后自动推进 `index_version`。
- 集成测试覆盖 current、modified、new、deleted、refresh 后恢复 current、current 时跳过 refresh。

未在本阶段实现：

- Watchdog 后台常驻命令。
- Git diff fast path。
- QueryService/RCAService 的强制 freshness gate。
- Evidence Report 的 file hash 绑定与 `validate_evidence`。
- symbol/chunk hash 级 embedding 复用。

## 验收标准

最小验收标准：

1. 初次索引后 freshness 为 `current`。
2. 修改已索引文件后 freshness 为 `stale`，并列出 changed file。
3. 新增 Python 文件后 freshness 为 `stale`，并列出 new file。
4. 删除已索引 Python 文件后 freshness 为 `stale`，并列出 deleted file。
5. `refresh_if_stale()` 能让 stale workspace 恢复为 `current`。
6. `refresh_if_stale()` 对 current workspace 不重复索引。
7. stale 刷新后 `index_version` 必须递增。

长期验收标准：

1. Query/RCA 不允许默认读取 stale index。
2. Evidence Report 能被独立校验 current/stale。
3. 大仓库增量刷新优先走 git diff fast path，并以 manifest hash 兜底。
4. Watchdog 只负责提前刷新，不负责最终一致性判定。
5. embedding 成本通过 symbol/chunk hash 复用降低，而不是依赖高风险 AST diff 作为第一阶段方案。

## 2026-07-02 实现状态更新

本阶段继续补齐动态更新入口：

- `QueryService` 支持 `project_root` 与 `auto_refresh`，每次 `search()`、`lookup_symbol()`、`get_symbol_info()`、`get_call_graph()` 前都会尝试 `refresh_if_stale()`。
- `RCAService` 在 `collect_trace_evidence()` / `analyze_trace()` 前执行 freshness gate，避免默认读取 stale index。
- `QueryResult` 与 `RCAResult` 增加 `snapshot` 字段，绑定 `index_version`、`freshness_status` 与证据文件 hash。
- `EvidenceReporter` 的 Markdown/JSON 报告会输出 snapshot 元数据。
- MCP 新增：
  - `repomind.refresh_index`
  - `repomind.validate_index_freshness`
  - `repomind.validate_evidence`
  - `repomind.changed_files_since`
- CLI 新增 `repomind watch`，默认使用 `watchdog` 文件系统事件触发 freshness gate 自动刷新索引；当 watchdog 不可用时回退到轮询，支持 `--once` 用于脚本化检查。
- `IndexService.changed_files_since()` 提供 git working tree Python 文件变更发现；该能力只作为 fast dirty-file detector，不替代 manifest hash 校验。
- `IndexService.validate_file_hashes()` 支持对 Evidence Report 绑定的文件 hash 进行 current/stale 校验。

仍未实现或暂不实现：

- symbol/chunk hash 级 embedding 复用；现阶段仍以文件级增量重建为正确性优先策略。
- Function-level AST diff；该项保留为后续性能优化，不作为一致性前置条件。
