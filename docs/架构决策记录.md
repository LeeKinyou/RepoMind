# 架构决策记录 (ADR)

本文档记录 RepoMind 项目中的关键技术选型决策及其理由。

---

## ADR-001: 选择 Tree-sitter 作为语法解析引擎

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 需要一个高性能的 Python 语法解析器，能够快速提取 AST 中的类、函数、导入等符号信息。

### 决策

采用 **py-tree-sitter** 作为主解析引擎。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Tree-sitter** | 纳秒级解析、语法容错、增量解析 | 缺乏强类型绑定 |
| **Jedi** | 类型推断能力强 | 解析速度较慢，不适合全量扫描 |
| **Pyright** | 完整类型推断 | 需要完整编译环境，部署复杂 |
| **ast (内置)** | 零依赖 | 性能差，不支持增量解析 |

### 理由

1. **性能**：Tree-sitter 基于 C 语言实现，解析速度比 Python 内置 ast 快 10 倍以上
2. **容错性**：即使代码有语法错误，也能提取出有价值的 AST 节点
3. **增量解析**：支持增量更新，只重新解析变更部分
4. **生态**：预编译的语法文件（tree-sitter-languages）开箱即用

### 后果

- 需要自行实现类型推断来弥补 Tree-sitter 缺乏类型绑定的不足
- 与 Jedi 配合使用，形成双引擎架构

## ADR-002: 选择 SQLite + NetworkX 而非 Neo4j

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 需要存储代码符号和调用关系的图结构，并支持高效的图遍历和算法计算。

### 决策

采用 **SQLite（磁盘持久化）+ NetworkX（内存计算）** 的组合方案。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SQLite + NetworkX** | 零配置、单文件、本地优先 | 不支持分布式 |
| **Neo4j** | 强大的 Cypher 查询、GDS 算法库 | 部署繁重、需 JVM、内存占用大 |
| **PostgreSQL + Apache AGE** | SQL + 图查询 | 需要安装数据库服务器 |

### 理由

1. **零配置**：SQLite 是 Python 内置的，无需额外安装
2. **本地优先**：单文件存储，可随项目一起复制
3. **性能**：NetworkX 的 PageRank、BFS 等算法在微秒级完成
4. **可移植性**：`.repomind/index.db` 可以直接拷贝到其他机器使用

### 后果

- 大型项目（>100 万节点）可能面临内存压力
- 不支持分布式部署（符合 MVP 的本地优先定位）
- 未来可通过轻量级扩展引入 Cypher 查询支持

## ADR-003: 选择 LanceDB 作为向量存储

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 需要存储代码的 Embedding 向量，支持语义相似度检索。

### 决策

采用 **LanceDB** 作为向量存储引擎。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **LanceDB** | 无服务器、磁盘存储、嵌入式 | 社区较小 |
| **ChromaDB** | 社区活跃、API 简洁 | 内存占用较大 |
| **FAISS** | Facebook 出品、性能优异 | 需要自行管理持久化 |
| **Milvus** | 功能强大、支持分布式 | 部署复杂，不适合 CLI 工具 |

### 理由

1. **无服务器架构**：与 SQLite 类似，无需后台服务
2. **磁盘存储**：向量存储在磁盘上，不占用内存
3. **Python 原生**：纯 Python API，易于集成
4. **混合检索**：支持向量 + 标量过滤的混合查询

### 后果

- 需要关注 LanceDB 的版本兼容性
- 大规模向量（>100 万）的检索性能需要验证

## ADR-004: 采用渐进式类型推断而非完整编译

**状态**：已采纳

**日期**：2026-06-20

### 背景

Python 是动态类型语言，需要推断变量和函数的类型信息来构建准确的调用图。

### 决策

采用 **渐进式多策略级联推断**，从高置信度到低置信度逐步尝试。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **渐进式推断** | 轻量、无需编译、覆盖多种场景 | 准确率不如完整类型检查 |
| **完整类型检查（Pyright/mypy）** | 准确率高 | 需要完整依赖环境，速度慢 |
| **纯 AST 分析** | 零依赖 | 无法处理动态类型 |

### 理由

1. **开箱即用**：不需要用户配置虚拟环境或安装项目依赖
2. **渐进增强**：有 Type Hints 就用 Type Hints，没有就尝试推断
3. **可控的准确率**：通过置信度分数让用户了解推断的可靠程度
4. **速度**：比完整类型检查快 10 倍以上

### 推断策略级联

| 优先级 | 策略 | 置信度 | 说明 |
|--------|------|--------|------|
| 1 | 显式类型提示 | 0.95 | `def f(x: int) -> str` |
| 2 | Import 映射 | 0.90 | `from models import User` |
| 3 | self 参数 | 0.85 | `def f(self)` → 类型为所在类 |
| 4 | 赋值推断 | 0.70 | `x = User()` → x: User |
| 5 | Jedi 推断 | 0.60 | Jedi 静态分析 |
| 6 | 鸭子类型 | 0.40 | `obj.save()` → 搜索有 save 方法的类 |

### 后果

- 无类型提示的代码推断准确率约为 70%（目标值）
- 需要维护六种推断策略的实现
- 用户可以通过添加 Type Hints 来提高推断准确率

## ADR-005: 采用 Explorer-Solver 双 Agent + 线性管道

**状态**：已采纳

**日期**：2026-06-20

### 背景

面对复杂的仓库级任务（如根因分析），需要 LLM Agent 的参与。但多 Agent 自由对话容易陷入死循环和 Token 浪费。

### 决策

采用 **Agentless 式线性管道 + Explorer-Solver 双 Agent** 的混合架构。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **线性管道 + 双 Agent** | 确定性强、避免死锁、Token 可控 | 灵活性较低 |
| **多 Agent 自由协商** | 灵活、能处理复杂场景 | Token 浪费、容易死循环 |
| **纯 Agentless** | 极简、Token 最省 | 缺乏深度推理能力 |

### 理由

1. **确定性**：线性管道确保流程可预测，不会无限循环
2. **关注点分离**：Explorer 专注搜索，Solver 专注推理
3. **上下文隔离**：Solver 的上下文中只有精炼的 Evidence Bundle，没有搜索噪音
4. **Token 效率**：避免多 Agent 之间的冗余对话

### 架构

```
输入 → [Explorer] → Evidence Bundle → [Solver] → 输出
         ↑                                │
         └──── Self-Refine Loop ──────────┘
```

### 后果

- 流程相对固定，灵活性不如多 Agent 自由协商
- 需要精心设计 Explorer 和 Solver 的 Prompt
- MVP 阶段足够，未来可扩展为更复杂的 Agent 编排

## ADR-006: 选择 LiteLLM 作为 LLM 接口封装

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 需要调用 LLM 进行根因分析和修复建议生成，需要支持多种模型提供商。

### 决策

采用 **LiteLLM** 作为 LLM 接口统一封装。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **LiteLLM** | 统一接口、支持 100+ 模型、一行切换 | 额外依赖 |
| **直接调用 OpenAI SDK** | 无额外依赖 | 只支持 OpenAI 兼容 API |
| **LangChain** | 功能丰富 | 过于庞大，依赖复杂 |

### 理由

1. **统一接口**：`litellm.completion()` 一行代码切换任何模型
2. **本地模型支持**：原生支持 Ollama、vLLM 等本地部署
3. **轻量**：相比 LangChain，依赖更少
4. **流式输出**：原生支持流式响应

### 后果

- 需要关注 LiteLLM 的版本更新
- 部分高级功能（如 Function Calling）可能需要额外适配

## ADR-007: 选择 Typer 作为 CLI 框架

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 需要一个用户友好的命令行界面。

### 决策

采用 **Typer** 作为 CLI 框架。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Typer** | 类型注解驱动、自动生成帮助、与 Click 兼容 | 额外依赖 |
| **Click** | 成熟稳定、社区大 | 需要更多样板代码 |
| **argparse** | 内置、无依赖 | API 繁琐 |
| **Fire** | Google 出品、极简 | 自定义能力弱 |

### 理由

1. **类型注解驱动**：函数签名即 CLI 定义
2. **自动生成帮助**：从 docstring 生成 `--help` 信息
3. **Rich 集成**：与 Rich 终端美化库无缝配合
4. **测试友好**：提供 `CliRunner` 用于测试

### 后果

- 需要维护类型注解的准确性
- 复杂的子命令嵌套需要额外设计

## ADR-008: 选择 Ruff 作为代码格式化和 Lint 工具

**状态**：已采纳

**日期**：2026-06-20

### 背景

需要统一的代码风格和质量检查工具。

### 决策

采用 **Ruff** 替代 Black + Flake8 + isort。

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Ruff** | 极快（Rust 实现）、All-in-one | 较新 |
| **Black + Flake8 + isort** | 成熟稳定 | 三个工具，配置分散 |
| **autopep8** | 轻量 | 功能有限 |

### 理由

1. **速度**：比 Flake8 快 10-100 倍
2. **All-in-one**：格式化 + Lint + Import 排序，一个工具搞定
3. **兼容性**：兼容 Black 格式、Flake8 规则、isort 排序
4. **VS Code 集成**：官方扩展支持

### 后果

- 团队成员需要安装 Ruff
- `.ruff.toml` 配置需要维护

## ADR-009: MVP 阶段仅支持 Python

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 的长期目标是支持多语言代码仓库分析。

### 决策

MVP 阶段 **仅支持 Python**，通过适配器接口预留多语言扩展能力。

### 理由

1. **时间约束**：1 周 MVP 周期，多语言会分散精力
2. **类型推断复杂性**：Python 动态类型已经够复杂
3. **生态熟悉度**：团队对 Python 生态最熟悉
4. **适配器模式**：已设计 `LanguageAdapter` 接口，未来可扩展

### 后果

- 需要明确在文档中标注"仅支持 Python"
- 适配器接口需要设计得足够通用

## ADR-010: 采用 Docker 沙箱作为默认安全方案

**状态**：已采纳

**日期**：2026-06-20

### 背景

RepoMind 的 RCA 功能会生成并执行代码，需要确保安全性。

### 决策

默认使用 **Docker 沙箱** 执行测试和验证代码，提供子进程模式作为备选。

### 备选方案

| 方案 | 安全性 | 性能 | 依赖 |
|------|--------|------|------|
| **Docker 沙箱** | ⭐⭐⭐⭐⭐ | 中 | Docker |
| **gVisor 沙箱** | ⭐⭐⭐⭐ | 中 | gVisor |
| **子进程模式** | ⭐⭐ | 高 | 无 |
| **不隔离** | ⭐ | 最高 | 无 |

### 理由

1. **安全性**：容器隔离提供最强的安全边界
2. **可配置**：内存限制、CPU 限制、网络禁用等
3. **易用性**：Docker Desktop 在各平台广泛可用
4. **备选方案**：Docker 不可用时可降级到子进程模式

### 后果

- 用户需要安装 Docker
- 首次运行需要拉取镜像
- 沙箱执行比直接执行慢 1-2 秒
